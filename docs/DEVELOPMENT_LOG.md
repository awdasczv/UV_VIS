## 2. 계산 엔진 선정

플랫폼별 배포 여부를 **추측이 아니라 실제 조회**로 확인했다.

| 도구 | Windows | 확인 방법 |
|---|---|---|
| Psi4 1.11 | ✅ | conda-forge win-64 빌드 존재 |
| xTB 6.7.1 | ✅ | conda-forge win-64 |
| PCMSolver 1.2.3 | ✅ | psi4 win-64 패키지의 의존성에 포함 |
| PySCF | ❌ | 공식 문서: "Windows 미지원, WSL 필요" |
| CREST | ❌ | 리눅스 바이너리만 배포 |
| DDX (ddCOSMO) | ❌ | conda-forge `pyddx` 에 win-64 없음 |

### 문서를 넘어선 확인

Psi4 공식 문서는 PCM + TD-SCF 를 *"RHF/UHF 에 대해 지원"* 이라고만 적고 있어
TD-**DFT** + PCM 은 문서상 미검증 영역이었다. 직접 시험한 결과:

| | S1 | S2 | S3 |
|---|---|---|---|
| 기체상 TDA (B3LYP/STO-3G, H₂O) | 11.714 | 13.976 | 14.648 eV |
| PCM TDA (동일 조건) | 11.982 | 14.167 | 14.820 eV |

값이 실제로 달라지므로 PCM 이 TD-DFT 에 반영된다. **WSL 없이 Windows 네이티브로
요구사항을 모두 만족할 수 있다는 결론.**

## 3. 마주친 문제들과 해결

### 3.1 에놀 구조가 킬레이트 고리를 못 만듦 — 구조 오류

**증상**: 에놀 A/B 에서 O–H···O 거리가 5.3 Å / 4.8 Å. 6원 고리가 닫히지 않음.

**원인 두 가지**
1. SMILES 이중결합 입체 지정이 반대였다.
   `/C(O)=C\C(=O)` 는 아릴기와 케톤 탄소를 **cis** 로 놓는데, 그러면 에놀 OH 와
   케톤 C=O 가 **trans** 가 되어 분자내 수소결합이 물리적으로 불가능하다.
2. MMFF94s 는 분자내 수소결합을 과소평가해 열린 s-trans 형태를 선호했다.

**해결**
1. SMILES 를 `/C(O)=C/C(=O)` 로 수정.
2. embedding 후 킬레이트 고리의 회전 가능한 이면각 두 개
   (Cenol–Cα–Cket–Oket, Cα–Cenol–Oenol–H) 를 0°로 명시 세팅 →
   구속 MMFF → 자유 MMFF 의 2단계 최적화.
3. 킬레이트가 실제로 닫힌(H···O < 2.2 Å) 구조만 후보로 채택.

**결과**: O–H···O = 1.76 Å (MMFF), GFN2-xTB 최적화 후 1.67 Å, O···O = 2.57 Å.
문헌의 CAM-B3LYP 값(1.55 Å)과 부합.

### 3.2 `/dev/null` 열기 실패 — 플랫폼 문제

`psi4.core.be_quiet()` 이 유닉스 경로 `/dev/null` 을 연다. Windows 에는 없다.
→ `psi4.core.set_output_file(<경로>, append)` 로 대체.

### 3.3 `-D3BJ` 분산보정 실패 — PATH 문제

`Program s-dftd3 is registered with QCEngine, but cannot be found`.
`s-dftd3.exe` / `dftd4.exe` 는 환경의 `Library\bin` 에 있었으나, conda 환경을
activate 하지 않고 `python.exe` 를 직접 호출해서 PATH 에 없었다.
→ `scripts/run.ps1` 래퍼가 PATH 를 세팅하도록 함.

### 3.4 xtb 호출에서 stdout 이 None — 로케일 문제 ★

**증상**: `TypeError: unsupported operand type(s) for +: 'NoneType' and 'str'`.
그 직전 별도 스레드에서 `UnicodeDecodeError: 'cp949' codec can't decode byte 0xe2`.

**원인**: 한국어 Windows 의 기본 로케일이 cp949 다. `subprocess.run(text=True)` 만
주면 파이썬이 xtb 출력의 UTF-8 박스문자(U+2500 등)를 cp949 로 디코딩하려다 실패하고,
출력을 읽는 리더 스레드가 죽어 `stdout` 이 `None` 이 된다.

**중요한 점**: xtb 계산 자체는 **정상 종료(returncode 0)** 였다.
로그를 읽지 않고 "계산이 실패했으니 이론 수준을 낮추자"고 판단했다면 원인을 영영
찾지 못했을 사례다.

→ `encoding='utf-8', errors='replace'` 명시.

### 3.5 PCM 이 17배 느리다는 오판 ★

처음에 첫 Davidson 반복 시간(440초)을 반복당 비용으로 보고 "PCM 이 기체상 대비
17배 느리다"고 결론지었다. **틀렸다.** 첫 반복에는 공동(cavity) 표면 자료구조를
만드는 일회성 비용이 포함된다. 총 시간으로 다시 재면:

| 조건 | SCF | TD-DFT (5상태) | λmax | f |
|---|---|---|---|---|
| 기체상 | 76 s | 149 s | 315.7 nm | 1.012 |
| PCM(에탄올) Area 1.0 | 325 s | 566 s | 332.3 nm | 1.141 |

**실제로는 약 3.8배.** 충분히 감당 가능한 수준이었고, 이 정정으로 계산 계획이
크게 바뀌었다.

그리고 결과가 문헌과 잘 맞았다:

| | 본 계산 | 문헌 (B3LYP/6-31+G(d), ACS Omega 2026) |
|---|---|---|
| 용매 적색이동 | +16.6 nm | +18.0 nm |
| 진동자 세기 | 1.012 → 1.141 | 0.991 → 1.138 |

### 3.6 PCM 격자 밀도 수렴 시험

`Cavity Area`(표면 조각 하나의 넓이)를 바꿔 수렴을 확인했다.

| Area (Å²) | 표면 조각 수 | SCF | TD-DFT | λmax |
|---|---|---|---|---|
| 1.0 | 1024 | 325 s | 566 s | 332.28 nm |
| 2.0 | 750 | 202 s | 425 s | 332.35 nm |

차이가 **0.07 nm** 에 불과하므로 격자 밀도에 대해 수렴했다.
Psi4 예제의 기본값 0.3 (조각 2149개) 은 이 목적에 불필요하게 촘촘하다.
→ 기본값을 1.0 으로 채택.

### 3.7 컨포머가 6개밖에 안 생성됨

ETKDG 의 `pruneRmsThresh = 0.5` 가 200개 요청을 6~7개로 잘라냈다.
RDKit 의 이 가지치기는 **수소를 포함한 전체 원자 RMSD** 를 쓰고 대칭을 고려하지
않아 지나치게 공격적이다.
→ 가지치기를 끄고, 중복 제거는 xTB 최적화 **후에** 제대로 한다.

### 3.8 가짜 중복 컨포머 ★

디케토에서 "고유" 컨포머 25개가 나왔는데 에너지 차가 0.02~0.03 kcal/mol 이었다.
tert-butyl 세 메틸기의 교환, 페닐 고리 180° 뒤집기, 거울상 배치를 단순 Kabsch
RMSD 가 서로 다른 구조로 판정한 것이다.

→ RDKit `GetBestRMS` (분자 그래프의 자기동형사상을 모두 시도해 최소 RMSD 를 찾음)
로 교체.

## 4. 컨포머 선택 전략을 바꾼 이유 ★

대칭 인식 중복제거 후에도 문제가 남았다. 대표 3개의 누적 볼츠만 가중치가
30~38% 밖에 안 됐다. 요구사항은 90~95% 였다.

**원인을 파고든 결과**: 컨포머들의 에너지가 거의 같아서 가중치가 고르게
퍼진 것이었다. 그런데 발색단(공액계) 비틀림각을 재보니:

| 토토머 | 고유 컨포머 | 발색단 기하 종류 |
|---|---|---|
| 에놀 A | 9 | **1** (아릴 비틀림 22–27°, 메톡시 0–2°) |
| 에놀 B | 11 | **1** (아릴 비틀림 15–29°) |
| 디케토 | 26 | **3** (골격 비틀림 70° / 19° / 76°) |

즉 에놀형 컨포머들의 차이는 **tert-butyl 메틸기 회전처럼 스펙트럼과 무관한
자유도뿐**이었다. 상위 3개만 뽑아 38% 를 덮는다는 것은 "앙상블의 62% 를
놓쳤다"는 뜻이 아니다. 빠진 컨포머의 스펙트럼이 뽑힌 것과 같기 때문이다.

**→ 선택 기준을 볼츠만 순위에서 발색단 기하 클러스터로 바꿨다.**

| 토토머 | 대표 수 | 커버 가중치 |
|---|---|---|
| 에놀 A | 1 (+검증용 1) | 100% |
| 에놀 B | 1 (+검증용 1) | 100% |
| 디케토 | 2 | 94% |

계산 수는 9개에서 6개로 줄었고 커버리지는 오히려 올라갔다.
각 에놀에는 "클러스터 안에서는 정말 스펙트럼이 같은가"를 실증하기 위해
발색단 기하가 가장 다른 구조를 하나씩 추가했다.

## 5. 하드웨어 조사 결론

사용자가 더 빠른 데스크탑(Ryzen 5 5600X + RTX 5060 Ti 16 GB)으로 옮길지
검토하면서 조사한 내용.

### GPU 는 기대만큼 도움이 되지 않는다

양자화학 계산은 전부 배정밀도(FP64)로 돈다. 소비자용 GPU 는 FP64 가 크게 제한된다.

| | FP64 | A100 대비 |
|---|---|---|
| A100 80GB (벤치마크 하드웨어) | 9.7 TFLOP/s | 1× |
| RTX 5060 Ti 16GB | 약 370 GFLOP/s (FP32 의 1/64) | 0.038× |
| Ryzen 5 5600X (CPU) | 약 375 GFLOP/s | — |

**5060 Ti 의 FP64 이론성능이 5600X CPU 와 거의 같다.** GPU4PySCF 가 A100 에서
보인 20배 같은 수치는 이 카드에 적용되지 않는다. RTX 50 시리즈에서 이 코드를
측정한 벤치마크는 존재하지 않고, Blackwell 커널 최적화 미완성 이슈가 열려 있다.
현실적 기대치는 3–8배(미검증).

### 정리

* Ryzen 5 5600X 는 이 노트북 대비 **약 3배** (PassMark 21,828 vs 9,317,
  게다가 노트북은 장시간 계산에서 12–15 W 로 열제한이 걸린다).
* **ORCA 6.1 이 가장 효과적인 다음 수단.** GPU 를 전혀 쓰지 않지만
  (6.1 매뉴얼에 "GPU"·"NVIDIA" 0회 등장) RIJCOSX 근사와 성숙한 CPCM/SMD 구현
  덕분에 같은 CPU 에서 훨씬 빠르다. Windows 네이티브 설치 파일 존재.
  단 orcaforum 가입 후 수동 다운로드 필요.

## 6. MINDO/3 비교 기준값에 대하여

프로젝트의 목표 중 하나가 "기존의 저비용 MINDO/3–TDA 계산보다 높은 수준"이었다.
그 비교 기준을 찾기 위해 문헌을 두 차례 조사했다.

**결론: 아보벤존에 대해 발표된 MINDO/3–TDA λmax 는 존재하지 않는다.**

Google Scholar 전문검색 기준:

| 질의 | 결과 |
|---|---|
| `"MINDO/3" avobenzone` | 0건 |
| `"MINDO/3-TDA"` | 0건 |
| `"MINDO/3" sunscreen "UV absorber"` | 0건 |

MINDO/3 는 본래 바닥상태 SCF 방법이며, 들뜬상태를 얻으려면 별도의 CI 를 붙여야
한다. 그렇게 아보벤존에 적용한 사례를 찾지 못했다.

**다만 실제로 존재하는 반경험적 아보벤존 UV-Vis 계산을 하나 찾았다**:
Tahir, Wijaya, Ahmadi (2006, UGM) — PM3 구조 최적화 + ZINDO/S,
에탄올 1분자를 붙인 명시적 초분자 모형. 저자들은 적색이동이 나타나지만
"실험값에 유의미하게 근접하지는 않았다"고 서술했다. 원문 PDF 가 로그인 장벽과
404 로 막혀 수치는 인용할 수 없어, 정성적 결론만 사용한다.

→ **동급의 저비용 기준선(HF/STO-3G + CIS)을 직접 계산해 비교 대상으로 삼는다.**
   보고서에는 "문헌에 MINDO/3 값이 존재하지 않음"을 검색 범위와 함께 명시한다.

## 7. 이 프로젝트의 작업 원칙

1. **실패하면 로그부터 읽는다.** 이론 수준을 낮추는 것은 원인을 분류한 다음이다.
   (3.4 의 인코딩 문제, 3.5 의 비용 오판이 이 원칙의 근거다.)
2. **추측하지 말고 잰다.** 플랫폼 지원 여부, 기저함수 개수, 계산 시간,
   격자 수렴은 전부 실제로 실행해서 확인했다.
3. **모든 실패는 분류해서 기록한다.** `logs/failures.json` 에 증상·원인·조치·결과.
4. **긴 계산은 단위별로 즉시 체크포인트를 남긴다.** 중간에 죽어도 처음부터
   다시 하지 않는다.
